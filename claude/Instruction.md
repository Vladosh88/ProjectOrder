1. Цель
Создать адаптивный веб-сайт для одного пользователя (фотографа) для ведения учёта выполненных заказов на фото товаров: создание, просмотр, редактирование, удаление заказов, загрузка фото/видео, Kanban-доска с двумя статусами, поиск и фильтрация, экспорт данных.

2. Функциональные требования (MVP + дополнения)
2.1. Авторизация
Защита доступа – Basic Auth на уровне хостинга фронтенда (Vercel).
Логин и пароль задаются через переменные окружения при деплое (один пользователь).
Бэкенд – без JWT, все API открыты, но доступны только после прохождения Basic Auth на фронте.

2.2. Главная страница – Kanban-доска
Две колонки: «В работе» (статус = 0) и «Готово» (статус = 1).
Карточки заказов: название, стоимость, дата выполнения, первое фото (миниатюра), индикатор просрочки (если срок прошёл и статус не выполнен).
Drag-and-drop карточек между колонками → меняет статус заказа (асинхронный запрос к API).
Плавные анимации (fade, scale).
Кнопка «Создать заказ» – модальное окно.

2.3. Карточка заказа (просмотр / редактирование)
Открывается по клику на карточку (модальное окно).

Поля:
Код заказа
Название (обязательное, до 255 символов)
Описание (текстовая область)
Галерея файлов (загруженные фото/видео) – превью, увеличение по клику, удаление отдельных файлов, добавление новых.
Срок выполнения (datetime-local)
Стоимость (число, рубли, два знака), рядом Стоимость +4% для расчета налога.
Статус (переключатель: выполнено / не выполнено)
Кнопки: «Сохранить» (валидация), «Удалить заказ» (с подтверждением), «Дублировать заказ» (копия без файлов, статус «Не выполнен»), «Закрыть».

2.4. Создание заказа
Модальное окно с полями: номер, код заказа, название, описание, загрузка файлов (можно несколько), срок, стоимость, рядом стоимость +4% для расчета налога.
Статус по умолчанию = 0 (не выполнен). 
Кнопка «Создать» – отправка данных, добавление карточки в колонку «В работе».

2.5. Удаление заказа
Только по одному: через карточку (кнопка «Удалить») или по иконке корзины на карточке в Kanban.
Подтверждение: «Вы уверены?».

2.6. Поиск и фильтрация
Поиск по названию (нечувствительный к регистру).
Код заказа

Фильтры:
По статусу (все / в работе / готово).
По просрочке (только просроченные).
Фильтры и поиск работают без перезагрузки страницы.

2.7. Пагинация
В каждой колонке – кнопка «Загрузить ещё» (по 20 карточек за раз).
Запросы к API с параметрами offset, limit, status.

2.8. Экспорт данных
Кнопка на главной – экспорт всех заказов в CSV и PDF (содержит поля заказа + URL-адреса файлов в Cloudinary).

2.9. Тёмная тема
Переключатель в шапке, сохраняется в localStorage.

2.10. Уведомления (опционально)
При загрузке страницы, если есть просроченные заказы – тост «Есть просроченные заказы».
Браузерное уведомление при разрешении пользователя.


3. Нефункциональные требования
Адаптивность: десктоп (две колонки), планшет (сжимаются), мобильный телефон (вертикальный список со сменой статуса через select).
Дизайн: в стиле Wildberries – белый фон, карточки с тенью, скругление 12px, акцентный цвет #9B4DFF, шрифт Inter, плавные переходы 0.2s.
Анимации: появление карточек, перетаскивание, открытие модалок.
Валидация: название не пустое; стоимость >0; срок не раньше сегодня; файлы – до 20 шт., до 50 МБ каждый, форматы: jpg, jpeg, png, webp, mp4, mov.
Обработка ошибок: тосты с сообщениями, блокировка кнопок на время отправки.

4. Стек технологий
Frontend
React 18 + Vite
Tailwind CSS
Zustand (или React Context)
React Hook Form + Zod
Axios
@dnd-kit/core + @dnd-kit/sortable (drag-and-drop)
react-datepicker
react-image-gallery (или своя галерея)
Деплой на Vercel с Basic Auth (через vercel.json)

Backend
Node.js + Express
PostgreSQL (управляемый, Supabase или Neon)
Prisma ORM (миграции, клиент)
Multer не используется – файлы загружаются напрямую на Cloudinary с фронтенда (виджет или их API).
Cloudinary SDK для подписи загрузки (опционально) и для получения URL готовых файлов.
Деплой бэкенда на Render (Web Service).
Хранилище файлов
Cloudinary – бесплатный тариф: 25 ГБ хранилища, 25 кредитов/мес на операции загрузки.
Схема: фронтенд запрашивает у бэкенда подписанные параметры для загрузки (или использует неподписанную загрузку с разрешёнными форматами).
В БД сохраняется public_id и secure_url из Cloudinary.
База данных (PostgreSQL)
Supabase (бесплатно) или Neon.
Таблицы: orders, files, tags, order_tags (см. схему в конце).

5. API Endpoints (бэкенд)
Метод	URL	Описание
GET	/api/orders?status=&search=&offset=&limit=	Список заказов с фильтрацией и пагинацией
GET	/api/orders/:id	Получить один заказ со списком файлов
POST	/api/orders	Создать заказ (поля + массив files: {public_id, url, format, ...})
PUT	/api/orders/:id	Обновить заказ (поля, обновление списка файлов)
DELETE	/api/orders/:id	Удалить заказ и удалить файлы из Cloudinary (через API)
POST	/api/orders/:id/duplicate	Дублировать заказ (без файлов)
GET	/api/export/csv	Экспорт CSV
GET	/api/export/json	Экспорт JSON
GET	/api/upload-signature	(опционально) получить подпись для загрузки на Cloudinary
6. Схема базы данных (Prisma)
prisma
model Order {
  id          Int      @id @default(autoincrement())
  title       String
  description String?
  price       Decimal? @db.Decimal(10,2)
  deadline    DateTime @db.Timestamptz
  status      Int      @default(0)
  createdAt   DateTime @default(now()) @map("created_at")
  updatedAt   DateTime @updatedAt @map("updated_at")
  files       File[]
  tags        Tag[]    @relation("OrderTags")
}

model File {
  id          Int      @id @default(autoincrement())
  orderId     Int      @map("order_id")
  publicId    String   @map("public_id")   // Cloudinary public_id
  url         String   @map("url")          // secure_url
  format      String?   // jpg, png, mp4...
  size        Int?      // в байтах
  createdAt   DateTime @default(now()) @map("created_at")
  order       Order    @relation(fields: [orderId], references: [id], onDelete: Cascade)
}

model Tag {
  id     Int     @id @default(autoincrement())
  name   String  @unique
  orders Order[] @relation("OrderTags")
}

model OrderTags {
  orderId Int @map("order_id")
  tagId   Int @map("tag_id")
  @@id([orderId, tagId])
  @@map("order_tags")
}
7. Условия деплоя на бесплатном хостинге
Frontend (React): Vercel.
Защита Basic Auth через vercel.json (логин/пароль в переменных окружения проекта).
Переменные окружения: VITE_API_BASE_URL (URL бэкенда на Render).
Backend (Node.js): Render Web Service.
Подключение к репозиторию GitHub, команда npm start.
Переменные окружения: DATABASE_URL, CLOUDINARY_CLOUD_NAME, CLOUDINARY_API_KEY, CLOUDINARY_API_SECRET.
База данных: Supabase (PostgreSQL).
Получить строку подключения (DATABASE_URL) из настроек проекта.
Выполнить миграции через Prisma.
Cloudinary: создать аккаунт, получить cloud name, API key, secret.
Настроить unsigned upload preset (для загрузки с фронта без подписи) или использовать signed uploads через бэкенд.

8. Критерии готовности (для проверки)
Пользователь заходит на URL, видит окно авторизации Basic Auth.
После ввода пароля – открывается Kanban-доска с двумя колонками.
Можно создать заказ с фото/видео (загрузка через Cloudinary).
Карточки перетаскиваются между колонками, статус меняется.
При клике на карточку открывается модалка с возможностью редактировать все поля, добавлять/удалять файлы.
Работает поиск по названию и фильтры по статусу/просрочке.
Работает экспорт в CSV/JSON.
Есть тёмная тема.
Адаптивная вёрстка корректна на ширине 320px, 768px, 1200px.
Нет ошибок в консоли, загрузка файлов работает без перезагрузки страницы.

10. Примечания для разработчика
Не нужно реализовывать регистрацию / сброс пароля.
При удалении заказа – удалять файлы из Cloudinary (вызов их API).
При дублировании – не копировать файлы, создавать пустой массив files.
Для загрузки файлов на фронтенде использовать Cloudinary Upload Widget (проще всего) или их REST API с подписью от бэкенда.
Обязательно добавить обработку ошибок при нерабочем Cloudinary или БД.
Использовать httpOnly для cookie не требуется – Basic Auth достаточно.
